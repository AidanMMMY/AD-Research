# iOS App 产品设计方案

**日期**：2026-06-23  
**项目**：ETF 量化投研平台  
**范围**：将现有 Web 平台完整移植为 iPhone + iPad Universal App  
**技术方向**：原生 SwiftUI，最低支持 iOS 16+

---

## 1. 背景与目标

现有平台是基于 FastAPI + React 的 ETF 量化投研系统，包含 16 个 Web 功能模块：Dashboard、ETF 目录、ETF 详情、筛选器、评分排行、组合池、组合详情、收益对比、相关性分析、行业轮动、策略管理、回测列表、回测详情、信号面板、报告中心、通知配置。当前 Web 端为桌面优先设计，没有移动端适配。

本设计目标：为自用/小团队场景打造原生 iOS App，保留专业分析深度，同时针对移动场景重新组织信息架构和交互方式。

---

## 2. 设计决策摘要

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 目标用户 | 自用 / 小团队 | 保留专业术语和复杂分析能力，不刻意简化 |
| 功能范围 | 完整移植 16 个模块 | 用户明确选择完整移植，后续分阶段实现 |
| 设备 | iPhone + iPad Universal App | 同时覆盖移动查看和大屏分析 |
| 离线策略 | 在线优先，少量缓存 | 元数据、收藏、组合列表本地缓存；详情实时拉取 |
| 技术栈 | 原生 SwiftUI | 体验上限最高，iPad 大屏和图表性能最好 |
| 最低系统 | iOS 16+ | 可用 Swift Charts，降低 K 线外第三方库依赖 |
| 视觉风格 | 深色专业主题 + iOS 控件 | 护眼、与 Web 品牌一致、符合 iOS 规范 |

## 3. 信息架构与导航

### 3.1 iPhone：底部 Tab Bar（5 个一级入口）

| Tab | 包含模块 |
|-----|---------|
| 首页 | Dashboard、自选关注、组合快捷入口、今日信号 |
| 市场 | ETF 目录、筛选器、评分排行、行业轮动 |
| 组合 | 组合池、收益对比、相关性分析 |
| 信号 | 信号面板、策略管理、回测中心 |
| 更多 | 报告中心、通知配置、设置 |

每个一级入口内部使用分段控制器（Segmented Control）或二级导航切换子功能。

### 3.2 iPad：左侧 Sidebar + 右侧 Detail

Sidebar 平铺所有模块，类似 iPadOS 系统应用：

- 首页
- ETF 目录 / 筛选器 / 评分排行 / 行业轮动
- 组合池 / 收益对比 / 相关性分析
- 信号面板 / 策略管理 / 回测中心
- 报告中心
- 通知 / 设置

支持分屏浏览、多窗口（Stage Manager）、列表-详情分栏。

## 4. 核心页面移动化适配

### 4.1 通用设计原则

- **首屏摘要优先**：每个页面顶部只放最关键的 2-3 个指标，详细内容向下滚动
- **图表即入口**：K 线、雷达图、饼图支持点击、长按、手势交互
- **Tab 切换替代并排**：Web 的左右分栏改为手机上分段控制器或顶部 Tab
- **手势增强**：图表支持捏合缩放、左右滑动切换周期、长按显示十字线

### 4.2 典型页面

| 页面 | 移动端设计 |
|------|-----------|
| 首页 Dashboard | 统计卡片 + Top 10 ETF 列表 + 我的组合 + 今日信号 |
| ETF 详情 | 顶部价格 + K 线图 + 周期切换 + 雷达图评分 + 技术指标卡片 |
| 组合详情 | 组合名称 + 资产配置饼图 + Tab 切换（持仓/相关性/快照）+ 再平衡提醒 |
| 筛选器 | 底部 Sheet 筛选条件 + 结果列表 |
| 回测详情 | 关键指标卡片 + NAV 曲线 + 交易记录列表 |

## 5. iPad 大屏适配

iPad 不应该是放大版的 iPhone，需专门设计：

- **分栏浏览**：ETF 列表与详情同屏，减少页面跳转
- **多窗口 / Stage Manager**：可同时打开多个 ETF 或组合页面对比
- **指针与悬停**：支持触控板悬停预览、右键菜单、指针形状变化
- **键盘快捷键**：⌘1-⌘5 切换模块，⌘F 搜索，⌘R 刷新
- **拖放**：从 ETF 列表拖拽 ETF 到组合池，快速调整持仓
- **更高信息密度**：同屏展示 K 线 + 雷达图 + 指标面板

## 6. 图表方案

| 图表类型 | 推荐方案 | 说明 |
|---------|---------|------|
| K 线/蜡烛图 | TradingView iOS SDK 或 DGCharts | 需支持缩放、十字线、MA/BB/RSI/MACD 叠加 |
| 折线/面积图 | Swift Charts | NAV 曲线、收益对比 |
| 柱状图 | Swift Charts | 行业轮动、评分排行 |
| 饼图/环形图 | Swift Charts | 资产配置 |
| 雷达图 | 自定义 SwiftUI Canvas | 五维评分，灵活控制 |
| 热力图 | 自定义 SwiftUI Grid | 相关性矩阵，颜色映射 + 点击显示数值 |

K 线优先使用 TradingView iOS SDK，因为其数据格式与现有 lightweight-charts 接近，迁移成本最低。

## 7. 数据层与缓存策略

### 7.1 技术栈

- **网络层**：`URLSession` + 自定义 `APIClient`
- **认证**：JWT 存储在 Keychain
- **本地缓存**：Core Data（兼容 iOS 16）或 SwiftData（如最低版本提升至 iOS 17）
- **后台刷新**：`BGAppRefreshTask`

### 7.2 在线优先策略

- 打开页面先展示缓存（如有），同时拉取最新数据
- 列表页缓存 1 小时，详情页缓存 15 分钟，信号/评分缓存 5 分钟
- 用户主动下拉刷新时忽略缓存
- 网络失败时展示缓存 + 顶部弱网提示条

### 7.3 错误处理

- 401：清理 Keychain，跳转登录页
- 500/超时：自动重试 1 次，仍失败展示错误页面
- 无网络：使用本地缓存并提示用户

## 8. 认证、安全与推送

| 能力 | 方案 |
|------|------|
| 登录认证 | JWT + Keychain 存储 |
| Token 刷新 | 过期前 5 分钟静默刷新 |
| 生物识别 | Face ID / Touch ID 解锁，敏感操作二次验证 |
| 推送通知 | Apple Push Notification Service（APNS） |
| 后台刷新 | 每日开盘前静默拉取数据 |
| 数据安全 | HTTPS、不使用 UserDefaults 存敏感信息 |

### 8.1 推送策略

- 信号通知：默认只推 BUY/SELL，不推 HOLD
- 再平衡提醒：权重偏离阈值超过 5% 时推送
- 报告通知：报告生成完成后静默通知

后端需扩展现有 webhook 模块，增加 APNS 推送能力。

## 9. 视觉设计系统

采用 **C 方案：深色专业主题 + iOS 控件**。

- 主背景：深色（接近 Web 端 #070b14 / #0f172a）
- 卡片：iOS 风格大圆角，分层背景色
- 强调色：蓝色（与 Web 端一致，如 #3b82f6）
- 涨跌色：与 A 股习惯一致，红涨绿跌
- 字体：SF Pro 系列，遵循 iOS 动态字体
- 控件：使用 SwiftUI 原生控件，保持 iOS 交互习惯

不采用玻璃拟态作为主导，避免 iOS 上的可读性问题；用纯色卡片 + 细腻阴影建立层级。

## 10. 实施阶段规划

| 阶段 | 周期 | 目标 | 关键交付 |
|------|------|------|---------|
| Phase 1：基础 + 核心监控 | 4-6 周 | 跑通 App，覆盖最高频场景 | 登录、Dashboard、ETF 列表/详情/搜索、自选、基础 K 线、Universal 骨架（iPhone 优先打磨，iPad 可运行） |
| Phase 2：组合与市场分析 | 4-6 周 | 覆盖投研核心分析能力 | 组合池、收益对比、相关性、行业轮动、资产配置图、iPad Sidebar |
| Phase 3：策略与回测 | 3-4 周 | 接入量化策略能力 | 策略管理、回测列表/详情/图表、信号面板、APNS 推送 |
| Phase 4：报告与打磨 | 2-3 周 | 补齐模块 + 准备上线 | 报告中心、通知配置、设置、iPad 高级适配、审核准备 |

### 10.1 关键决策点

- Phase 1 先验证 iPhone 核心体验，Phase 2 再补 iPad
- Phase 3 需要后端增加 APNS 发送逻辑
- Phase 4 的报告中心可先通过 WebView 加载现有 HTML 报告，后续再原生化

## 11. 风险与假设

| 风险 | 应对措施 |
|------|---------|
| K 线图实现复杂度高 | 优先使用 TradingView iOS SDK，避免自研 |
| iPad 分屏/多窗口开发成本高 | Phase 2 再启动，Phase 1 专注 iPhone |
| 后端 APNS 推送需要改造 | Phase 3 之前完成后端扩展 |
| App Store 审核对金融类应用要求严格 | 避免提供交易功能，定位为投研工具 |
| 完整移植工作量超出预期 | 严格按阶段交付，每阶段可独立上线 |
